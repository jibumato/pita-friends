# ピタフレ サービス開始までの進行表

現状（コアフロープロトの実装完了）から、実サービスとしてのローンチまでのロードマップです。
時期は固定日程ではなく **フェーズ（マイルストーン）と依存関係** で表現しています。チーム規模・外部ベンダー選定・法務確認の進み方で前後します。

## 設計原則: 女性ファースト安全設計

「女性と遊びたい男性の需要が高い」という非対称を前提にしつつ、**供給が少なく離脱しやすい側（女性）を最優先で守る**ことで、結果的に需要（男性）を満たす持続可能なモデルを採用します。女性が安心できないと供給が枯れ、マッチが成立しなくなるためです。ピタフレ既存の安全思想（本人確認・マナースコア・外部誘導/金銭要求の禁止・出会い目的の通報）と一致します。

具体化の柱：

- **女性のコントロール** — 誰から誘い/連絡を受けるか（本人確認済みのみ／同性のみ／全員）、誘いの承認制、オンライン・検索表示の公開範囲を本人が決められる（`安心設定`）
- **承認制の受信** — 誘いはリクエストとして届き、相手の信頼情報（本人確認・マナー・ドタキャン率・共通ゲーム）を見て承認するまでトーク・連絡先は開かない（`受け取った誘い`）
- **安全宣言** — 出会い目的ではなくゲームを一緒に楽しむ場であることを明示。低マナー・未確認ユーザーからの接触を制限
- **募集の参加条件** — 全員／同性のみ／女性のみ、本人確認済み限定

> 男性が女性を一方的に探しやすくする設計ではなく、女性の同意・安全・コントロールを土台に据えることで、健全な男女比とマッチ成立率を両立させます。

## マネタイズ: コイン制マーケットプレイス（GameRoom型）

[game-room.jp](https://game-room.jp/) を参考に、**コインで「一緒に遊ぶ時間」を売買するマーケットプレイス**を採用しました（フロントエンド実装済み）。

- **コインウォレット** — 1コイン≒1円でパック購入（300 / 1,000+ボーナス / 3,000+ボーナス / 6,000+ボーナス）
- **ホスト機能** — ユーザーは任意で「ホスト」として掲載し、時給コインを設定して一緒に遊ぶ時間を提供できる
- **予約** — 「さがす」でホストの時給を見て、30分/1時間/2時間の予約 → コイン消費 → 確定でマッチフローへ接続。残高不足時はチャージ導線を提示
- **安全文言の整合** — 既存の「金銭のやり取り禁止」は**「アプリ外・直接の金銭要求は禁止、公式コイン決済は運営が仲介する安全な支払い」**に書き換え済み（Safety系画面・README）
- **出会い系該当性リスクの低減（法務レビュー反映済み）** — 募集の「女性のみ」条件を削除し受付は防御的な「全員／同性のみ」に限定、募集はゲーム必須を明示、さがすの検索結果から性別バッジを削除、通報カテゴリに「出会い・恋愛目的の勧誘」を追加、ホストのルールに「提供するのはゲーム同行の時間・恋愛/出会い目的の勧誘は禁止」を明文化
- **運用面の法務対応（運用Q&A反映済み）** — オンボーディングに**「みまもり」同意画面**を追加（メッセージ監視の目的・方法・範囲を独立画面で明示し明示的同意を取得、通信の秘密対応）、本人確認画面に**画像原本の照合後削除**を明示、予約画面に**帰責事由別のコイン返還ルール**を明示、通報対応の文言を努力義務表現に修正

> ⚠️ **法務確認が必須（Phase 5で対応）**: (1) コイン購入は前払式支払手段にあたり得るため**資金決済法**の届出・供託義務を確認。(2) 「異性との交際希望者への紹介に金銭が絡む」構成は**出会い系サイト規制法**の適用可能性があるため、実態（ゲーム同行が主目的であること）の整理と年齢確認体制（本人確認は実装済み）の確認。(3) ホストへの分配・出金がある場合は資金移動業/収納代行の該非確認。これらはローンチ前に専門家レビューが必要です。詳細な論点整理は [`docs/legal/coin-economy-legal-review.md`](docs/legal/coin-economy-legal-review.md) を参照。

## Phase 2 — モバイル基盤 🚧

- ✅ 配信形態の決定：**PWA化を採用**（依存なしで manifest + service worker）。ホーム画面に追加してスタンドアロン起動、オフラインでもアプリシェルが動作
- ✅ 端末ストレージ / セッション永続化 — ユーザー設定（テーマ / 性別 / 安心設定 / コイン残高 / ホスト設定）を `localStorage` に保存・復元（`src/persist.ts`）。既知キーのみ検証してマージし、壊れたデータは無視。アカウント・トークン等はバックエンド接続時（Phase 3）
- ⬜ プッシュ通知（合流リマインド等）※ バックエンド + Web Push 必要
- ⬜ カメラ連携（本人確認の書類・顔写真撮影）※ 本人確認フロー実装時

> 残る2項目はいずれもバックエンド（Phase 3）または本人確認フロー（Phase 4）に依存するため、フロントエンド単体でこれ以上進められる項目はここまで。

## Phase 3 — バックエンド & データ 🚧

> バックエンドは **Supabase**（Postgres + 認証 + リアルタイム + ストレージ）を採用。接続手順は [`docs/backend-setup.md`](docs/backend-setup.md) を参照。環境変数未設定でも既存のモックデモは変わらず動作する。

- ✅ DB 設計（ユーザー / 安心設定 / 本人確認 / コイン経済 / ホスト / マッチング・約束 / レビュー・通報・ブロック）— `supabase/migrations/0001〜0005` にRLS込みで実装済み。マナースコア・コイン残高はクライアントから直接書き込めない設計（`SECURITY DEFINER`関数経由のみ）
- ✅ 認証（サインアップ・ログイン・セッション）— `signUp`/`signIn`画面を追加し、ウェルカム画面から実際にSupabase Authへ結線。バックエンド未接続時は従来どおりデモ動線（PRESS START→安心設定、ログイン→ホーム）を維持。設定画面にログアウト導線を追加
- ✅ プロフィール・安心設定・ホスト設定・コイン残高・信頼スタッツの実データ配線 — サインイン/サインアップ時とアプリ起動時のセッション復元時に`hydrateAccount()`でSupabaseから取得し、ローカル状態を上書き。ニックネーム・性別・安心設定・ホスト設定の変更はSupabaseへ書き込み（楽観的更新+失敗時ロールバック）。マイページの「ログアウト」（従来無反応だった）も結線
- ✅ コイン購入・ホスト掲載の未実装ガード — コイン購入は決済代行未選定のため`purchase_coins`をクライアントから呼べない設計。バックエンド接続時はウォレット画面で「決済連携は準備中です」を表示しパックを無効化。ホスト掲載は本人確認必須（DBトリガーで強制）のため、失敗時は分かりやすい日本語メッセージを表示しローカル状態をロールバック
- ⬜ さがす画面・予約の実データ化 — ホスト一覧（`host_settings`+`profiles`+`profile_trust_stats`）を実際に検索・予約できるようにするのは未着手（現状はモック3人のまま）。マッチング/相性スコアの算出ロジックが未設計のため、実装時は表示項目の再設計が必要
- ⬜ よく遊ぶゲーム・プレイスタイル（`profiles.favorite_games`/`play_style`）の実データ配線 — セッション内の「今回選択中のゲーム」(`state.game`)と概念が異なるため、今回は意図的に見送り
- ⬜ 本人確認（eKYC）— `identity_verifications`テーブルはあるが、Verify画面は引き続き完全モック。ベンダー連携なしでは実際の確認ができないため、実装するとホスト掲載が原理的に不可能になる（正しい挙動）
- ⬜ セッション管理の高度化（トークン自動更新はSupabase任せだが、ログアウト時のリアルタイム購読解除等は未検証）
- ⬜ リアルタイム基盤（チャット・オンライン状態）— WebSocket（Supabase Realtime想定）
- ⬜ マッチングエンジン（相性スコア算出ロジック）
- ⬜ 通知配信基盤
- ⬜ `promises`（約束）のライフサイクル遷移（joined/completed・フレンドコード開示）— テーブル設計のみ、状態遷移関数は未実装

> ⚠️ **重要な修正**: `package.json`の`lint`スクリプトが`tsc --noEmit`（ルートの`tsconfig.json`は`"files": []`のプロジェクト参照のみで実質何もチェックしない）だったため、このセッションの型チェックが機能していなかったことが判明。`tsc -b`に修正し、あわせて`src/lib/database.types.ts`の型不備（postgrest-jsが要求する`Relationships`/`Views`フィールド欠落により、クエリの戻り値が軒並み`never`になっていた）も修正。今後は`npm run build`（`tsc -b && vite build`）が実質的な型チェックのゲート。

## Phase 4 — 安全・信頼（コア差別化） 🚧

ピタフレの中核価値。ここの完成度がサービスの信頼を決める。

> 実装設計は [`docs/trust-safety-spec.md`](docs/trust-safety-spec.md) にドラフト済み（算出式・しきい値・フロー図）。バックエンド未接続のため、実装自体はPhase 3以降。

- 🚧 マナースコア / 信用スコア / ドタキャン率の算出・反映 — 算出式・表示ルール・コールドスタート対策を設計済み
- 🚧 通報 → 運営審査 → 利用制限のオペレーションフロー — 5段階フロー・緊急度分類・措置マトリクス・異議申立てを設計済み
- 🚧 外部アプリ誘導・金銭要求の検知（NGワード / パターン検知） — 検知カテゴリ・誤検知対策・エスカレーション基準を設計済み
- 🚧 本人確認（eKYC）ベンダー選定・統合 — 選定基準（画像非保持要件を含む）を設計済み、ベンダー比較は未着手
- 🚧 年齢確認（18歳未満の排除） — 登録時の機械判定フローと事後発覚時の対応手順を設計済み
- ⬜ ブロック機能 — データモデル・効果範囲を設計済み、UIの3導線化（ブロックのみ/通報のみ/両方）は未着手
