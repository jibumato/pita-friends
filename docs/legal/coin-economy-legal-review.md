ピタフレに、**ユーザーがコインを購入し、「一緒に遊ぶ時間」を提供するホストに時間単位でコインを消費する**マーケットプレイス機能を実装しました（[game-room.jp](https://game-room.jp/) を参考）。**資金決済法・出会い系サイト規制法・資金移動業/収納代行・特定商取引法**の4領域について確認したい事項があります。質問文一式は [6章「質問リスト」](#6-弁護士行政書士への質問リストq-a形式) にまとめています。

## 2. 現状のピタフレの実装内容（正確な仕様）

コードベース参照: `src/flow.ts`, `src/screens/Wallet.tsx`, `src/screens/HostSettingsScreen.tsx`, `src/screens/Booking.tsx`

### 2.1 コイン購入
- 1コイン ≒ 1円換算
- 販売パック: 300コイン(¥300) / 1,000+50コイン(¥1,000) / 3,000+300コイン(¥3,000) / 6,000+900コイン(¥6,000)
- 決済手段: **未実装**（フロントエンドのモックのみ。実際の決済代行事業者は未選定）
- コインの返金・出金・失効ルール: **未定義**（要設計）

### 2.2 ホスト機能
- ユーザーは任意で「ホスト」として掲載登録でき、**時給（1時間あたりのコイン数）**を自分で設定できる（既定400コイン/時、50〜2,000コインの範囲で調整可）
- 掲載条件: 本人確認済みユーザーのみ（既存の本人確認フローに依拠、実装済み）
- 掲載時に対応ゲーム・自己紹介文を設定
- 性別によるフィルタは設けていないが、既存の「安心設定」（同性のみ／本人確認済みのみ等）は掲載中も有効という建て付け

### 2.3 予約・決済
- 利用者は「さがす」画面でホストの時給を見て、**30分／1時間／2時間**の単位で予約
- 予約確定時にコインを消費（`hourlyRate × 分数 ÷ 60` で算出、四捨五入）
- コインが不足している場合は予約不可、チャージ導線を提示
- **ホストへの対価の還元・出金の仕組みは未実装**（コインは減るだけで、ホスト側にコインや金銭が付与される導線は現状ない）
- 予約確定後は、既存の「誘う→マッチ」フローと同じ演出（返事待ち→マッチ）に接続。実際の合流・プレイ・レビューは既存のコアフローをそのまま利用

### 2.4 既存の安全設計との関係
- 「金銭のやり取り禁止」という既存の安全文言は、「**アプリ外・直接の金銭要求は禁止／公式コイン決済のみ運営が仲介する安全な支払い**」に書き換え済み
- 出会い目的の利用は引き続き禁止と明示（オンボーディング）
- 本人確認・マナースコア・通報機能は全ユーザーに適用

### 2.5 未実装（法務判断に影響する可能性がある領域）
- 決済代行事業者（Stripe等）の選定
- コインの利用規約（有効期限、返金不可の範囲、譲渡禁止等）の文面
- ホストへの報酬分配・出金フロー
- 年齢制限の運用（本人確認は実装済みだが、18歳未満の確実な排除フローは未検証）
- 特定商取引法に基づく表記ページ（ハンドオフ資料内に画面デザインは存在するが未実装）

## 3. 参考にしたサービス: GameRoom（game-room.jp）

**注記**: 以下は公開Webページ・検索結果から把握できた範囲の情報であり、GameRoom社の内部規約・登録状況の全容を確認したものではありません。実際の法令遵守状況の裏取りにはならないため、あくまで「ピタフレが模倣しようとしているビジネスモデルの外形」の参考情報として扱ってください。

- ゲーム好きの相手（ゲーマー）と、コインを介してオンラインで一緒にゲームをプレイできるマッチングサービス
- ユーザーはサービス内通貨（コイン）を購入し、ゲーマーが設定した時給（例: 1時間300〜500コイン等）に応じてコインを消費して予約する
- 利用規約上、退会時にサービス内通貨・チケットの払い戻し・返還は行わない旨の記載がある（＝コインは返金不可という設計）
- 運営が「金銭のやり取りを仲介することで安全性を担保する」という文脈でコイン決済を位置づけている（＝直接の金銭授受は禁止し、コイン決済に一本化する設計思想）

ピタフレのコイン経済は、この「コイン一本化による安全性担保」というロジックをそのまま踏襲しています。

## 4. 検討が必要な法令と論点

### 4.1 資金決済法（前払式支払手段）

**論点**: ピタフレのコインは、円で購入し、アプリ内（＝発行者自身が提供するサービス内）でのみ使用できる設計です。これは一般に「**自家型前払式支払手段**」に該当し得ます（第三者が発行する加盟店向けの「第三者型」とは区別されます）。

自家型前払式支払手段に該当する場合、一般的な制度としては以下のような義務が生じ得ます（**要専門家確認・数値は要最新確認**）:

- 基準日（毎年3月31日・9月30日）時点の**未使用残高が一定額を超える場合の内閣総理大臣（財務局）への届出義務**
- 未使用残高の一部について**発行保証金の供託義務**
- 利用者への表示義務（有効期限、苦情窓口、譲渡制限の可否等）
- 資金決済法上の禁止行為（現金への払い戻しの原則禁止 等）

→ 質問は [6章 A](#a-資金決済法前払式支払手段) 参照。

### 4.2 出会い系サイト規制法（インターネット異性紹介事業に係る規制）

**論点**: この法律は「異性交際を希望する者の求めに応じて異性を紹介する」事業を「インターネット異性紹介事業」として都道府県公安委員会への届出制で規制しています。ピタフレは「ゲームを一緒に遊ぶ相手を探す」ことを目的として掲げていますが、以下の要素が事業の性質判断に影響し得ます:

- 性別を条件に相手を検索・フィルタできる機能がある（「安心設定」の同性のみ／全員 等、`src/flow.ts` の `ContactScope`）
- 個人間のメッセージ機能・通話機能がある
- 今回、**金銭を介して特定の異性と1対1の時間を過ごす**機能（ホスト予約）を追加した

→ 質問は [6章 B](#b-出会い系サイト規制法) 参照。

### 4.3 資金移動業／収納代行（ホストへの対価還元がある場合）

**論点**: 現状はホストへの対価還元・出金フローは未実装ですが、将来的に「ユーザーが支払ったコインの一部をホストに金銭として還元する」設計にする場合、**プラットフォームが利用者間の資金移動を仲介する事業者**とみなされる可能性があります。日本では近年、C2C型プラットフォームが実質的に「収納代行」の名目で個人間送金に近い機能を提供しているケースについて、金融庁が資金移動業規制の適用可能性を指摘する報告を出しており、規制動向が流動的な領域です。

→ 質問は [6章 C](#c-資金移動業収納代行) 参照。

### 4.4 特定商取引法

**論点**: コイン（デジタルコンテンツ・役務利用権）の有償販売にあたるため、通信販売として特定商取引法に基づく表示（事業者名・所在地・連絡先・返品ポリシー等）が必要になる可能性があります。ハンドオフのデザイン資料には該当画面（「20 特定商取引法に基づく表記」）が存在しますが、**未実装**です。

→ 質問は [6章 D](#d-特定商取引法) 参照。

### 4.5 参考: 適用可能性が低いと考えられるが確認したい論点

- **風俗営業等の規制及び業務の適正化等に関する法律（風営法）**: 物理的な店舗・接待を伴わない完全オンラインサービスのため、通常は対象外と考えられますが、念のため除外理由を明確化しておきたいです。
- **青少年保護育成条例（都道府県ごと）**: 未成年同士・成人と未成年の接触を助長しないための年齢確認の実効性について、本人確認フローの精度を含め確認したいです。

→ 質問は [6章 E](#e-その他確認) 参照。

## 6. 弁護士・行政書士への質問リスト（Q&A形式）

> 以下の**回答**は専門家からの暫定的な整理です。数値・要件は制度改正で変わり得るため、**最終判断・最新の基準は必ず弁護士・行政書士等の確認を受けてください**。

### A. 資金決済法（前払式支払手段）

**Q1.** 当社が発行するコイン（1コイン≒1円、パック購入制、アプリ内でのみ使用可能）は、資金決済法上の「自家型前払式支払手段」に該当しますか。

**回答**: かなり高い確率で該当します。「金銭を支払う→コインを取得→コインでサービスを受ける→他店では使えない」という構造は、資金決済法第3条の前払式支払手段の典型例です（Apple Storeのゲーム内通貨やLINEコインと同様の整理）。

**Q2.** 該当する場合、内閣総理大臣（財務局）への届出義務が生じる未使用残高の基準はどの程度ですか。

**回答**: 基準日（3月31日・9月30日）時点の**未使用残高が1,000万円を超えた場合**に届出義務が生じます。未使用残高＝販売額−利用額−失効額。

**Q3.** 発行保証金の供託義務は生じますか。生じる場合、供託額の算定方法を教えてください。

**回答**: 届出対象になると供託義務があります。供託額は**未使用残高の2分の1**です（例: 未使用残高3,000万円 → 1,500万円を供託）。

**Q4.** 「コインは退会時に払い戻さない」という設計は、資金決済法上・消費者保護の観点から問題ありませんか。

**回答**: 一般的には可能です。資金決済法13条で原則として払戻しは禁止されているため、「退会しても返金しない」という設計はむしろ通常です。ただし、購入前・利用規約・特商法表示などで十分に説明する必要があります（消費者契約法上、不意打ちにならないことが重要）。

**Q5.** 利用者への表示義務（有効期限、苦情窓口、譲渡制限等）として、具体的に何を明示する必要がありますか。

**回答**: 最低限、①発行者 ②コイン名称 ③有効期限 ④利用できるサービス ⑤利用できないケース ⑥払戻し不可 ⑦問い合わせ先 ⑧利用規約 を表示すべきです。届出後は法定表示事項が追加されます。

### B. 出会い系サイト規制法

**Q6.** 性別による相手の絞り込み機能（同性のみ／全員から選択）と、金銭を介して特定の1人と1対1の時間を予約する機能を併せ持つ場合、「インターネット異性紹介事業」に該当するリスクはどの程度ありますか。

**回答**: 現状の仕様では一定のリスクがあります（①1対1 ②特定相手を選ぶ ③時間予約 ④性別検索、の4要素が揃うため）。ただし、ゲームを一緒に遊ぶことだけが目的で、恋愛・性交目的の媒介でないのであれば、直ちに該当するとは言えません。**警察実務では「サービス名称より実態」で判断**されるため、広告・利用実態が重要です（完全一致の判例はなく、行政運用・風俗営業関連裁判例からの実態判断になります）。

**Q7.** 「ゲームを一緒に遊ぶことが主目的」という実態を維持しつつ、該当性を避けるためには、UI・規約・マーケティング上どのような線引きが有効ですか。

**回答**（重要度順）:
- ① **ゲームタイトル必須** — 「APEX募集」「VALORANT募集」等、募集はゲーム前提にする
- ② **遊ぶ時間だけ予約** — 「デート」「会う」等は禁止
- ③ **恋愛目的禁止** — 利用規約だけでなく通報対象にもする
- ④ **プロフィール** — 恋人募集／恋愛／結婚 等を書けないようにする
- ⑤ **検索軸** — ゲーム／ランク／VC／時間帯 のみ。性別検索は極力なくす、あるいは「表示しない」をデフォルトにする
- ⑥ **マーケティング** — 「異性とも遊べる」「かわいい女の子と」等の広告は絶対に出さない

**Q8.** 現在実装済みの本人確認フローは、この法律が求める年齢確認の水準を満たしていますか。

**回答**: eKYC・運転免許証・マイナンバーカード・在留カード等を確認しているのであれば、出会い系規制法上要求される年齢確認水準を満たす可能性があります。ただし、実際の本人確認フローを確認しないと断定はできません。

### C. 資金移動業／収納代行

**Q9.** 現状（ホストへの金銭的還元・出金機能なし）であれば、資金移動業規制の対象外と考えてよいですか。

**回答**: ホストへ出金しないのであれば、資金移動業ではないと考えてよいでしょう。お金が「運営→運営」で完結しているためです。

**Q10.** 将来的にホストへコインを金銭として還元・出金する機能を追加する場合、資金移動業登録が必要になりますか。

**回答**: ここは非常に重要な論点です。「利用者 → コイン購入 → ホスト → 現金化」という流れになると、**資金移動業または収納代行の検討が必要**になり、かなり慎重な設計が求められます。

**Q11.** 収納代行としての整理は可能ですか。可能な場合、どのような条件を満たす必要がありますか。

**回答**: 可能性はあります（ただし最高裁判例はなく、収納代行に関する金融庁Q&A・行政解釈からの類推になります）。運営が「ホストの代理で料金を回収しているだけ」という実態であれば、収納代行として整理できる余地があります。ただし、**運営の裁量で残高を管理する設計は危険**です。Stripe Connect等を使う設計が一般的です。

### D. 特定商取引法

**Q12.** コイン（デジタルコンテンツ・役務利用権）の販売にあたり、特定商取引法上どのような表示が必要ですか。

**回答**: 販売者／所在地／メールアドレス／電話番号／価格／支払方法／支払時期／提供時期／キャンセル（返品）ポリシー／動作環境 などの表示が必要です。

**Q13.** 個人開発・法人化前の段階でも、この表示義務は生じますか。

**回答**: 生じます。法人でなくても、個人事業主でも「販売者」として表示義務があります。

### E. その他確認

**Q14.** 完全オンライン・非対面のサービス形態であれば、風俗営業等の規制及び業務の適正化等に関する法律（風営法）の適用対象外と考えてよいですか。

**回答**: 現状、オンラインのみ・ゲームのみであれば、通常は風営法の対象外です。ただし、オンライン接客や疑似キャバクラのような形態になると別問題になります。

**Q15.** 都道府県の青少年保護育成条例の観点から、追加で必要な年齢確認・利用制限はありますか。

**回答**: 各都道府県で異なりますが、一般的には18歳未満の深夜利用・有害情報・異性交流などへの配慮が求められます。18歳未満の利用を許可するなら、保護者同意や夜間利用制限を検討した方が安全です。

### F. 総合

**Q16.** 上記を踏まえ、ローンチ前に対応すべき優先順位はどうなりますか。

**回答**（優先度順）:

| 重要度 | 対応内容 | 目安 |
|---|---|---|
| ★★★★★ | 出会い系該当性の排除（UI・広告・規約） | 最重要 |
| ★★★★★ | 利用規約・プライバシーポリシー・特商法表示 | 最重要 |
| ★★★★★ | 前払式支払手段としてのコイン設計・未使用残高管理 | 最重要 |
| ★★★★☆ | 本人確認・年齢確認体制 | 重要 |
| ★★★★☆ | 利用者保護（通報・監視・BAN体制） | 重要 |
| ★★★☆☆ | 供託・届出基準の管理 | 利用者増加後 |
| ★★☆☆☆ | 将来のホスト出金制度の法的設計 | 後回しで可 |

**総括**: 現時点の「ユーザーがコインを購入し、ホストへの金銭還元・出金機能がなく、ゲームを一緒に遊ぶ時間を予約する」という設計であれば、**最大の法的論点は資金決済法ではなく、出会い系サイト規制法上の実態評価**です。資金決済法については、自家型前払式支払手段としての管理を適切に行えば対応できる可能性が高い一方、サービスのUI・検索項目・広告表現・運営実態によっては、警察から「異性紹介を目的とするサービス」と評価されるリスクがあります。

## 7. 法務リスクを下げるための設計変更オプション（検討候補）

### 7.0 今回すでに実装した対応（出会い系該当性リスクの低減）

上記回答の最優先事項「出会い系該当性の排除（UI・規約）」を受け、フロントエンドに以下を反映済みです（`src/screens/BoardCreate.tsx` / `ReportSheet.tsx` / `HostSettingsScreen.tsx`）。

- **募集の「女性のみ」条件を削除** — 能動的に特定の性別を指定して募る導線をなくし、参加受付は「全員／同性のみ」の**防御的な受付制限**のみに限定（性別フィルタは「誰から誘い・連絡を受けるか」という受信側の安全設定としてのみ残す）
- **募集はゲーム必須を明示** — 「ゲーム（必須）」と表示し、ゲーム同行が前提であることを明確化
- **通報カテゴリに「出会い・恋愛目的の勧誘」を追加** — 規約だけでなく通報導線でも恋愛・出会い目的を排除
- **ホストのルールに目的の明文化** — 「提供するのは"ゲームを一緒に遊ぶ時間"。出会い・恋愛目的の勧誘は禁止」を追記

> なお、`さがす`画面には**性別による検索フィルタは元々存在しません**（検索軸はゲーム／プレイスタイル／本人確認のみ）。回答B7⑤の「性別検索は極力なくす」は現状すでに満たしています。ただし検索結果カードには性別バッジを表示しており、これを残すか否かは要判断です。

### 7.1 追加で検討すべきオプション

専門家の判断を待つ前提で、さらにリスクを下げる方向の設計オプションを検討候補として挙げておきます（**いずれも未決定・要相談**）。

- **A. 性別条件を「ホスト予約」機能から切り離す**: 予約機能では性別フィルタを使わせない（安心設定は既存のマッチングのみに適用し、ホスト検索は別ロジックにする）
- **B. コインの用途を人の時間売買以外にも広げる**: ブースト・アバター課金等、対人課金以外の用途を増やし、「人を検索して金銭を払う」という単一目的の色を薄める
- **C. ホストへの金銭還元を当面実装しない**: 現状のまま「消費されるだけ」の設計を維持し、資金移動業の論点が発生するタイミングを先送りする
- **D. 特定商取引法表記・利用規約・プライバシーポリシーを先に整備する**: 最低限の表示義務を先に満たす
