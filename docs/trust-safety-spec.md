# ピタフレ 安全・信頼機能 仕様書（Phase 4 実装設計）

**位置づけ**: 本資料はROADMAP Phase 4「安全・信頼（コア差別化）」の各項目を、バックエンド実装前に固めておく**設計仕様**です。フロントエンド（現状のモック表示）と整合する形で、実データ・実ロジックに置き換える際の仕様を定義します。運用上の法務論点は [`docs/legal/operations-legal-qa.md`](legal/operations-legal-qa.md) を参照してください（本資料はその実装レベルの詳細）。

作成日: 2026-07-16

## 1. マナースコア（信用スコア）

現状UIでの表示: `★4.9`〜`★5.0`（`Profile.tsx`・`Requests.tsx`の`TrustStat`・`Result.tsx`）。5点満点、小数点第2位まで表示。

### 1.1 算出式

```
manner_score = clamp(1.00, 5.00, base + review_component - penalty_component)
```

- **base**: 初期値 `4.50`（新規ユーザーの中立スタート。0スタートだと不当に不利、5スタートだと実績なしで最高評価になるため中間値）
- **review_component**: 直近**30件**のプレイ後レビュー（★1〜5）の加重平均。新しいレビューほど重みを大きくする指数減衰（半減期90日）。1件も無ければ0
- **penalty_component**: 確定した違反（通報が「違反あり」で確定したもの）1件につき `-0.15`〜`-0.50`（重大度に応じ§3.4のマトリクスに準拠）。ドタキャンは§2で別指標のため二重減点しない

### 1.2 表示ルール

- レビュー件数が**3件未満**の間は数値の代わりに「まだ実績が少ないメンバーです」というバッジを表示し、スコアで不利に見えないようにする（コールドスタート対策）
- スコアの算出根拠（直近レビュー・違反歴の有無）は本人のみ「マイページ」から確認できる（運用Q&A Q15の訂正請求導線と対応）
- 明らかな誤記録（レビューの誤操作等）は運営に申立て可能。申立て中は係争中フラグを表示し、確定するまでスコアに反映しない

### 1.3 更新タイミング

プレイ後レビューが送信された時点、および通報の判定が確定した時点でバッチ的に再計算（同期的な即時計算で十分な規模を想定）。

## 2. ドタキャン率

現状UIでの表示: `0%`（`Profile.tsx`・`Result.tsx`・`Requests.tsx`の`dotakyan`）。

### 2.1 定義

「ドタキャン」＝確定した約束（トークで確定した予定、またはコイン予約）を、**開始1時間前以降にキャンセル**、または**無断で欠席**したもの。これは`Booking.tsx`のキャンセルルール（開始1時間前までは無料キャンセル可、以降は原則キャンセル不可扱い）と対応する定義であり、UI文言と数値の意味を一致させる。

### 2.2 算出式

```
dotakyan_rate = (直近90日のドタキャン件数 / 直近90日の確定した約束の総数) × 100%
```

- 分母が**3件未満**の場合は「データなし」と表示し、0%と誤解される数値は出さない
- 90日のローリングウィンドウとし、改善したユーザーが古い記録に永久に縛られないようにする

### 2.3 コイン予約との関係

- ゲスト都合のドタキャン（開始1時間前以降のキャンセル/無断欠席）→ ドタキャン率にカウント。コインは`terms-of-service-draft.md`第9条の返還マトリクス通り扱う（ゲスト都合・直前は返還なし）
- ホスト都合のドタキャン（無断欠席・大幅遅刻）→ ホスト側のドタキャン率にカウントし、コインはゲストへ全額再付与（実装済み文言と整合）

## 3. 通報 → 運営審査 → 利用制限のオペレーションフロー

現状UIの通報カテゴリ（`ReportSheet.tsx`）: 外部アプリへの誘導 / アプリ外での直接の金銭・RMT要求 / 出会い・恋愛目的の勧誘 / 暴言・ハラスメント / なりすまし・年齢詐称
（`SafetyCenter.tsx`にはこれに加えて「ドタキャン・無断欠席」「その他」もあり）

### 3.1 フローの5段階

```
① 受付 → ② 一次トリアージ(自動) → ③ 人による審査(必要な場合) → ④ 判定・措置 → ⑤ 通知・記録
```

1. **受付**: 通報時点の対象メッセージ・プロフィール・予約情報をスナップショットとして保全（運用Q&A Q4・Q13に対応。保存期間は§3.5）
2. **一次トリアージ**: カテゴリと通報者・被通報者の過去実績から重大度を自動仮判定（§3.4）。「原則24時間以内に内容の確認に着手するよう努める」（`SafetyCenter.tsx`の実装済み文言、運用Q&A Q1の努力義務表現に準拠）
3. **人による審査**: 一次トリアージで「軽微」以外、または争いがある場合に実施。閲覧者・閲覧範囲・閲覧ログを記録（運用Q&A Q12）
4. **判定・措置**: §3.4のマトリクスに従い、警告／一時停止／強制退会を選択
5. **通知・記録**: 被通報者に理由の概要を通知（詳細を開示しない場合がある旨は規約に明記済み）。通報者には「対応しました」の完了通知のみ（詳細な処分内容は非開示）

### 3.2 緊急度の分類

| 区分 | 該当例 | 一次対応 |
|---|---|---|
| 緊急 | 生命・身体の危険が疑われる通報 | ただちに一時停止＋案内文で110番を促す（`SafetyCenter.tsx`の既存文言と対応） |
| 重大 | 金銭要求、出会い目的の勧誘、なりすまし・年齢詐称 | 24時間以内に人による審査へ回す |
| 軽微 | マナー違反、ドタキャンの申告 | 自動トリアージのみ。累積件数がしきい値を超えた場合のみ人による審査 |

### 3.3 累積しきい値（軽微カテゴリの自動エスカレーション）

同一ユーザーへの軽微カテゴリの通報が**90日以内に3件**累積した時点で、自動的に人による審査キューに上げる（個別の通報1件だけでは判断が難しい「マナー」系の通報を、パターンとして拾うための設計）。

### 3.4 措置マトリクス

| 累積違反回数 | 措置 | マナースコアへの影響 |
|---|---|---|
| 1回目（軽微） | 警告表示のみ | `-0.15` |
| 1回目（重大） | 一時利用停止（7日） | `-0.30` |
| 2回目（重大）、または緊急区分 | 一時利用停止（30日）＋再発防止の確認 | `-0.50` |
| 3回目、または悪質性が高い場合 | 強制退会＋身分証ベースの再登録防止（運用Q&A Q7） | — |

### 3.5 証跡の保存期間

運用Q&A Q13の回答に準拠し、違反関連ログは**6か月〜1年**、重大事案（詐欺等）は**事件終結まで**保存。

### 3.6 異議申立て

被通報者は、措置から**7日以内**に簡易な再審査を申し立てられる（運用Q&A Q3「簡易な再審査フロー」への対応）。再審査中は措置を維持したまま、担当者が再検討する。

## 4. 外部アプリ誘導・金銭要求の検知（NGワード・パターン検知）

「みまもり」への同意画面（`Consent.tsx`）で説明している「原則自動、人の目視は限定」を実装する一次検知ロジック。

### 4.1 検知カテゴリ

| カテゴリ | 検知パターン例 | 誤検知リスク |
|---|---|---|
| 外部連絡先 | LINE ID・電話番号らしき数字列・InstagramやX等のSNSハンドルのパターン、「ライン交換」等の語彙 | 中（ゲーム内フレンドコードと混同しやすい） |
| 金銭要求 | 「振込」「現金」「直接払う」「個人間で」等の語彙＋金額らしき数値の近接 | 低 |
| 出会い目的の勧誘 | 「会いたい」「彼女/彼氏募集」「タイプ」等の語彙 | 中（ゲームの文脈でも使われ得る） |

### 4.2 実装方式

1. **一次検知（自動・全件）**: キーワード辞書＋正規表現によるルールベース。ゲーム用語の許可リスト（例: 「フレンドID」「フレンドコード」はLINE誘導と誤検知しない）を維持し、誤検知を抑える
2. **本人への警告表示**: 初回ヒットはブロックせず、送信前に「外部への連絡先交換や金銭のやり取りはアプリ内で完結させてください」という確認ダイアログを出す（送信自体は本人の選択に委ねる。過検知による正常な会話の阻害を避ける）
3. **通報キューへのエスカレーション**: 同一ペアで**繰り返しヒット**した場合、または「金銭要求」カテゴリが**1回でも**ヒットした場合は、人による確認対象としてフラグを立てる（運用Q&A Q11・Q12の「自動処理が原則、人の目視は通報時・違反兆候の検知時のみ」に対応）
4. **対象範囲**: メッセージ・募集文・プロフィール文のみ。予約時のホスト時給等の定型情報は対象外

### 4.3 誤検知時の救済

自動検知により警告や制限を受けたユーザーは、§3.6と同じ異議申立て導線を使える。

## 5. 本人確認（eKYC）ベンダー選定基準

現状UIの2ステップ（`Verify.tsx`）: ①本人確認書類を撮影 → ②顔写真で本人照合。書類はマイナンバーカード・運転免許証等を想定。

### 5.1 選定の必須要件

- 運転免許証・マイナンバーカード・在留カードなど日本国内の主要書類に対応
- 顔写真との1:1照合（なりすまし対策）
- 書類から**生年月日を自動抽出**でき、18歳未満を登録時点で機械的に判定できる（§6と連動）
- **画像を長期保持しない**運用、または当社側で照合後にすぐ削除できるAPI設計（運用Q&A Q6の方針と整合。ベンダー側が結果のみを返し生画像を保持しない、または当社が能動的に削除APIを呼べることを要件とする）
- 日本の個人情報保護法に対応した委託契約が結べること（運用Q&A Q8の委託先監督義務を満たせること）

### 5.2 評価軸（候補選定時に個別評価）

精度（誤認率）／価格（月額・従量）／審査のリードタイム／画像非保持オプションの有無／サポート言語。具体的なベンダー名は選定フェーズで別途比較する。

## 6. 年齢確認（18歳未満の排除）

### 6.1 登録時の一次防御

`Verify.tsx`の本人確認完了時点で、eKYCが抽出した生年月日から年齢を機械的に判定。18歳未満の場合は**登録を完了させない**（`Consent.tsx`→`Verify.tsx`の途中で止め、Setup画面には進ませない）。

### 6.2 事後発覚時の対応

運用Q&A Q21・Q22に準拠したフロー:

1. ただちに利用停止
2. 法定代理人（保護者）への連絡・事実確認
3. 未使用コイン残高の返金（取消しの意思表示を待たず自主返金）
4. 使用済み分は現存利益の範囲で個別協議
5. 再登録防止目的での本人確認情報の最小限保持を通知
6. 対応記録を保存（出会い系サイト規制法の行政対応における防御資料にもなる）

## 7. ブロック機能

### 7.1 データモデル

```
Block { blockerId, blockedId, createdAt, reason? }
```

片方向の関係として保持する（Aがブロック＝Aの検索結果・通知からBが消える。Bからは通常のユーザーとして見えるが、Bからのメッセージ・誘いはAに到達しない）。

### 7.2 効果範囲

- 双方の「さがす」検索結果から相互に非表示
- ブロック対象からの新規メッセージ・誘い・予約リクエストを拒否
- 既存のトークルームは残すが新規メッセージ不可（証跡保全のため削除はしない）
- **相手に通知しない**（`SafetyCenter.tsx`の既存文言「相手に通知されません」と一致させる）

### 7.3 通報との関係

現状のUI（`ReportSheet.tsx`）は「ブロックして通報する」という単一の主導線だが、`SafetyCenter.tsx`には独立した「ブロックする」カードもある。実装時は次の3つの独立した操作として提供する:

- ブロックのみ（通報を伴わない、単純な関係遮断）
- 通報のみ（まだ関係は続けたいが運営に報告したい場合）
- ブロック＋通報（現状のデフォルト動線）

## 8. 今後の実装順序（目安）

1. §1・§2（スコア算出）と§7（ブロック）— 既存のモックデータ構造をそのまま実データ化できる、依存の少ない部分
2. §6（年齢確認）と§5（eKYCベンダー選定）— 本人確認フローの実装に必須で、他の全機能の前提になる
3. §3（通報オペレーション）— 人的リソース（審査担当）の体制構築とセットで進める
4. §4（NGワード検知）— メッセージ機能（リアルタイム基盤）の実装後
